# Shared Global Jobs Pool — Design

**Date:** 2026-06-14
**Status:** Approved (brainstorming), pending implementation plan
**Scope:** Backend data-model change + matching + minimal UI. One implementation plan.

## Goal

Turn the per-user `jobs` table into a single **shared global pool of UNSA convocatorias**.
A convocatoria scraped from one user's inbox is stored once (one row, one embedding) and
becomes a match candidate for every UNSA user. This makes the candidate pool larger and
matching cheaper (no re-embedding the same convocatoria N times). Email stays the MVP
ingestion channel; platform-native posting is future work, out of scope here.

Also: surface the already-stored `deadline` (fecha límite) in the feed UI.

## Background — current state

- `jobs` (`src/server/drizzle/schemas/jobs.ts`) is **per-user**: `user_id` NOT NULL FK,
  uniques `(user_id, gmail_msg_id)` and `(user_id, dedupe_hash)`. The same convocatoria
  received by N users produces N rows + N embeddings.
- `deadline` (`date`, nullable) already exists, is extracted by Gemini
  (`src/server/ai/extract-job.ts`), persisted, used in the dedupe hash, and already
  filtered against `CURRENT_DATE` in retrieval (`matching.ts`). **No new date column is
  needed** — the request is display-only.
- Dedupe hash (`src/server/services/dedupe.ts`) = `normalize(titulo) | normalize(empresa) |
  weekKey(deadline ?? msg.date)`, SHA-256. This already collapses the same convocatoria
  across inboxes (same title+empresa+week), so it works as a **global** identity unchanged.
- Ingestion (`src/server/services/ingestion.ts`): `ingestOneJob` builds a row scoped to the
  current `userId`, embeds, then `persistJob` does `INSERT ... ON CONFLICT DO NOTHING`.
  Noise emails are written as `jobs` rows with `is_job=false` + `noise_reason` (so the
  per-user gmail-msg unique stops reprocessing).
- Matching (`src/server/services/matching.ts`, `retrieveCandidates`): hard filters
  `eq(jobs.userId, userId)`, `eq(jobs.isJob, true)`, `isNotNull(jobs.embedding)`,
  vigencia `or(isNull(deadline), gte(deadline, CURRENT_DATE))`, optional location.
- `matches` (`src/server/drizzle/schemas/matches.ts`): `(userId, jobId)` pairs, unique
  `(userId, jobId)` — private per-user. Unchanged by this design.
- Migrations live in `drizzle/` (`0000`,`0001`,`0002`), generated by `drizzle-kit generate`
  from `src/server/drizzle/schemas/index.ts`. Next is `0003`.

## Architecture decision

Jobs become **public shared data** for the whole UNSA tenant. Convocatorias are mass emails
from `udeeg_convocatorias@unsa.edu.pe`, not private user content, so this is a deliberate,
scoped exception to the per-user isolation rule. **Profiles and matches remain private
per-user** (`where userId = session.user.id`), so no private data leaks: a user only ever
reads their own `matches` rows.

Per-user "which Gmail messages have I already processed" tracking moves out of `jobs` into a
new `ingested_messages` log. The global `jobs` table holds only real convocatorias.

## Data model changes

### `jobs` → global (`src/server/drizzle/schemas/jobs.ts`)

Remove (no longer meaningful for a global, deduped pool):
- `userId` column + its FK + `jobs_user_id_idx` index.
- `gmailMsgId` column (per-inbox, no global identity).
- `isJob` column (noise no longer enters `jobs`; pool is jobs-only).
- `noiseReason` column (moves to `ingested_messages`).
- uniques `jobs_user_gmail_msg_unique` and `jobs_user_dedupe_unique`.

Add:
- unique `jobs_dedupe_unique` on `(dedupeHash)` — global convocatoria identity.

Keep unchanged: `id`, `sourceSender`, `titulo`, `empresa`, `modalidad`, `ubicacion`,
salary columns, `requisitos`, `skills`, `deadline`, `applyLink`, `rawEmail`, `dedupeHash`
(now globally unique), `embedding` (vector 768) + HNSW index, `createdAt`.

Provenance (`first_seen_by` etc.) is intentionally **out of scope** (YAGNI).

### New table `ingested_messages` (`src/server/drizzle/schemas/ingested-messages.ts`)

Per-user processing log; replaces the old per-user gmail-msg uniqueness on `jobs`.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK `defaultRandom()` | |
| `userId` | text NOT NULL | FK → `user.id`, `onDelete: cascade` |
| `gmailMsgId` | text NOT NULL | |
| `jobId` | uuid NULL | FK → `jobs.id`, `onDelete: set null`; null when noise |
| `noiseReason` | text NULL | set when the email was classified as not-a-job |
| `createdAt` | timestamp NOT NULL `defaultNow()` | |

Constraints: unique `(userId, gmailMsgId)` (each user processes each email once),
index on `(userId)`.

Register both schema files in `src/server/drizzle/schemas/index.ts`.

### Migration `0003_*` + data reset

- Generate via `drizzle-kit generate` (drops the columns/uniques above on `jobs`, creates
  `ingested_messages`, adds the global dedupe unique).
- **Reset regenerable data:** truncate `jobs`, `matches`, `ingestion_runs` (and the new
  `ingested_messages` is empty). Approved by user — pre-deploy, data is regenerable via
  `scripts/seed-demo.ts` and live re-ingestion; this avoids a fragile collapse-and-remap
  migration. The truncate is a separate explicit step (SQL appended to the generated
  migration or a documented `db:reset` run before re-seeding), not silent.

## Ingestion changes (`src/server/services/ingestion.ts`)

New per-user flow:

1. List Gmail messages for the user (unchanged).
2. Batch-load existing `ingested_messages.gmailMsgId` for this user; **skip** messages
   already processed. (Replaces the old reliance on the `jobs` gmail-msg unique.)
3. For each new message: classify + extract (unchanged Gemini calls).
4. If **noise**: insert `ingested_messages(userId, gmailMsgId, jobId=null, noiseReason)`.
   Do **not** write a `jobs` row.
5. If **job**: compute `dedupeHash` (unchanged), then:
   - `INSERT INTO jobs (... WITHOUT embedding ...) ON CONFLICT (dedupe_hash) DO NOTHING
     RETURNING id`.
   - If a row was returned (newly inserted, convocatoria did not exist globally):
     **embed now** and `UPDATE jobs SET embedding = ? WHERE id = ?`. This is the cost/speed
     win — embedding happens once per unique convocatoria.
   - If no row returned (already existed, possibly inserted concurrently):
     `SELECT id FROM jobs WHERE dedupe_hash = ?` to get the existing `jobId`. **No embedding
     call.**
   - Insert `ingested_messages(userId, gmailMsgId, jobId)`.

`persistJob` is reshaped accordingly (insert-then-embed-if-new instead of embed-then-insert).
`ingestion_runs` metrics stay per-user; "jobs created" now counts globally-new jobs this run
contributed (define explicitly: rows where the insert returned an id).

`retrieveCandidates`'s `isNotNull(jobs.embedding)` already guards rows that are mid-embed, so
a brief window without an embedding cannot surface a broken candidate.

## Matching changes (`src/server/services/matching.ts`)

- `retrieveCandidates`: **remove `eq(jobs.userId, userId)`** and **remove `eq(jobs.isJob,
  true)`** (column dropped). Keep `isNotNull(jobs.embedding)`, vigencia, and optional
  location filters. Every user now retrieves from the whole pool.
- `getFeed` join query: drop any selection of removed columns (`userId`, `gmailMsgId`,
  `isJob`, `noiseReason`); it already filters by `matches.userId`, so it stays private.
- `matches` upsert target `(userId, jobId)` unchanged.
- **No fan-out.** A newly-pooled job enters a user's feed the next time that user
  ingests/refreshes and `runMatching` runs for them — matching against the full pool. This
  is the MVP; recomputing matches for all users on every new job is out of scope.

## UI change (`src/frontend/components/feed/match-card.tsx`)

- Show `deadline` when present: a small badge/line, e.g. "Cierra: 20 jun" (and a
  near/expired variant like "Vence hoy" when `deadline` is today/near). Display only —
  reuse the existing date formatter in `src/frontend/lib/format.ts` or add a pure
  `formatDeadline` helper there (unit-testable). No data/endpoint change; `deadline` already
  flows through the feed payload (confirm it's selected in `getFeed`; add to the select if
  missing).

## Testing

- `dedupe.ts`: existing tests stay green (hash unchanged); add a case asserting two
  different users / inboxes for the same convocatoria produce the **same** hash (global
  collapse).
- Ingestion service tests: (a) a message already in `ingested_messages` is skipped;
  (b) a convocatoria that already exists globally is **not** re-embedded and links the new
  user's `ingested_messages.jobId` to the existing job; (c) noise writes an
  `ingested_messages` row with `jobId=null` and no `jobs` row.
- Matching service tests: `retrieveCandidates` returns jobs regardless of who ingested them
  (no user filter); vigencia/location filters still applied.
- `format.ts`: unit-test `formatDeadline` (present / today / past / null).
- No DOM test runner — MatchCard verified via `pnpm check` + `pnpm build` + manual walk
  (consistent with prior specs).
- Update `scripts/seed-demo.ts` to the new schema (no `userId`/`isJob` on jobs; seed global
  jobs + per-user matches).

## Security / privacy

- `jobs` is shared public data (UNSA convocatorias). `rawEmail` is the same public
  convocatoria body; whoever ingests first sets it. No private user content in `jobs`.
- `profiles` and `matches` stay strictly per-user (`where userId = session.user.id`, never a
  client-supplied id). Feed reads only the requesting user's `matches`.
- Gmail scope stays read-only; no token/CV/raw-body/rerank-payload logging.

## Out of scope (YAGNI)

- Platform-native job posting (future channel that replaces email).
- Match fan-out / recompute-for-all-users on new job.
- Provenance columns (`first_seen_by`).
- Preserving/collapsing existing job/match data (user chose reset).
